# BEIMS 建筑能源智能管理系统用户使用手册

## 1. 系统概述

BEIMS (Building Energy Intelligent Management System) 是一款基于自然语言处理的建筑能源智能管理系统，旨在通过对话式界面让非技术人员也能便捷地查询和分析建筑能耗数据。

### 1.1 核心功能

- **数据层**：能耗数据集构建、清洗与标准化，支持CSV、Excel格式导入，数据验证和质量检查
- **查询统计模块**：基于MCP协议的数据接入与查询接口，支持按建筑、时间、监测参数等条件精准查询，13类核心统计分析功能，12种可视化图表展示，统计报表自动生成与导出（PDF格式）
- **智慧运维模块**：领域知识库构建，基于RAG技术的智能问答，能耗异常原因分析，设备运行状态查询，节能建议生成
- **系统集成**：RESTful API接口，MCP协议支持，Vue.js前端界面，响应式设计
- **智能交互**：自适应打字机效果，停止按钮（截断模式），智能澄清机制

### 1.2 技术栈

- **后端**：FastAPI、SQLAlchemy、PostgreSQL/SQLite、Pandas、NumPy、Plotly、Matplotlib、LangChain、ChromaDB、Sentence Transformers、ReportLab
- **前端**：Vue.js 3、Element Plus、ECharts、Axios、Pinia
- **AI技术**：Ollama (本地LLM)、通义千问 (云端LLM)、ChromaDB (向量数据库)

## 2. 系统架构

BEIMS系统由两个独立但相互协作的架构组成：**前后端网页系统架构**和**智能助手架构**。

### 2.3 各层路由机制

| 层级 | 名称 | 处理方式 | 响应时间 | 适用场景 | 示例 |
|------|------|---------|---------|---------|------|
| **Layer 0** | 知识库优先 | ChromaDB向量检索 | 100-300ms | 故障/操作/概念类 | "空调外机坏了怎么办" |
| **Layer 1** | 静态规则 | Python正则匹配 | **1-10ms** ⚡ | 结构化数据查询 | "XX建筑YY日的电耗" |
| **Layer 2** | 语义路由 | Sentence-BERT嵌入 | 200-500ms | 模糊意图匹配 | "帮我看看用电情况" |
| **Layer 3** | 本地推理 | Ollama qwen2.5:7b | 1-3s | 复杂SQL生成+分析 | "为什么上周能耗异常?" |
| **Layer 4** | 云端增强 | 通义千问API | 3-8s | 高级分析和长文本 | "生成年度节能报告" |

- **Python 3.8+**：用于运行后端服务

## 4. 智能助手相关文件功能说明

### 4.1 api_server.py - 智能助手API服务器

**功能**：
- 提供智能对话接口（POST /chat），支持上下文记忆和多轮对话
- 集成云边协同路由分发器，实现4层过滤逻辑
- 支持实时监测推送（WebSocket /ws/monitor）
- 提供建筑信息查询、能耗数据统计等API
- 实现本地模型（Ollama）和云端模型（通义千问）的无缝切换
- 集成知识库检索功能，支持运维、故障、操作类问题回答

**核心特性**：
- 上下文记忆：支持最多20轮对话历史
- 智能意图分析：自动识别用户查询意图
- SQL自动生成：根据自然语言查询生成SQL语句
- 知识库增强：结合ChromaDB向量数据库提供精准回答
- 多建筑支持：可查询14个不同建筑的能耗数据

### 4.2 cloud_edge_router.py - 云边协同路由分发器

**功能**：
- 实现4层云边协同路由机制，优化响应速度和准确性
- 硬规则过滤引擎：毫秒级响应高频查询
- 语义路由层：基于Sentence-BERT的意图匹配
- 本地推理层：使用Ollama qwen2.5:7b进行意图解析和SQL生成
- 云端增强层：处理复杂分析任务
- 安全失败保护机制：自动降级到本地模式

**核心特性**：
- 智能澄清机制：当输入模糊时自动引导用户明确需求
- 多模式支持：相对时间查询、趋势分析、对比分析等
- 断网容灾：云端服务不可用时自动降级到本地模式
- 性能优化：优先使用规则引擎处理简单查询，复杂查询交给LLM
- 可扩展性：支持自定义规则和意图模板

## 5. 智能助手问答示例

### 5.1 数据查询类

**示例1：基本数据查询**
- **用户**：Caspian 2021年7月21日的电耗是多少？
- **系统**：Caspian 2021年7月21日的电耗为 125.6 kWh（电耗）

**示例2：时间范围查询**
- **用户**：Michigan 2021年5月7日上午6点到8点的水耗
- **系统**：
  **水耗** 查询结果（共 2 条记录）：
  • 平均值：2.5 m³
  • 最小值：2.3 m³
  • 最大值：2.7 m³

**示例3：趋势查询**
- **用户**：Aral 2021年7月的能耗趋势
- **系统**：
  分析了 Aral 2021年7月的能耗数据，呈现以下趋势：
  • 月初（1-10日）：平均电耗 110 kWh/天
  • 月中（11-20日）：平均电耗 135 kWh/天，较月初增长 22.7%
  • 月末（21-31日）：平均电耗 120 kWh/天，较月中下降 11.1%
  • 整体趋势：7月电耗呈现先上升后下降的波动趋势

### 5.2 分析类

**示例1：异常分析**
- **用户**：为什么 Ontario 的能耗突然增加？
- **系统**：
  根据 Ontario 的能耗数据，能耗增加可能的原因包括：
  1. **空调系统负荷增加**：数据显示 HVAC 能耗占总能耗的 65%，当室外温度升高时，空调系统运行时间增加
  2. **人员密度变化**：人员密度从平均 0.5 人/100m² 增加到 1.2 人/100m²，导致用电设备使用增加
  3. **设备运行异常**：冷冻水供水温度与回水温度差增大，可能存在设备效率下降
  
  **建议**：
  • 检查空调系统运行状态，确保设备正常运行
  • 优化空调温度设置，建议夏季温度设置为 26°C
  • 实施智能照明控制，减少非工作时间的能耗

**示例2：对比分析**
- **用户**：对比 Caspian 和 Erie 的能耗情况
- **系统**：
  **Caspian vs Erie 能耗对比分析**：
  • **总电耗**：Caspian 为 12,560 kWh，Erie 为 10,240 kWh，Caspian 高 22.7%
  • **空调能耗占比**：Caspian 为 68%，Erie 为 55%，Caspian 空调能耗占比更高
  • **单位面积能耗**：Caspian 为 150 kWh/m²，Erie 为 120 kWh/m²
  • **能耗模式**：Caspian 峰值出现在下午 2-4 点，Erie 峰值出现在上午 10-12 点
  
  **分析结论**：
  - Caspian 的能耗水平整体高于 Erie，主要差异在于空调系统的使用
  - 建议 Caspian 优化空调系统运行策略，参考 Erie 的能耗模式

### 5.3 知识库类

**示例1：故障处理**
- **用户**：空调外机坏了怎么办？
- **系统**：
  **空调外机故障处理步骤**：
  1. **立即停机**：关闭空调电源，避免进一步损坏
  2. **检查电源**：确认电源连接正常，检查断路器是否跳闸
  3. **清洁检查**：检查外机散热片是否积灰，清洁散热片
  4. **检查制冷剂**：观察是否有制冷剂泄漏迹象（油污）
  5. **专业维修**：如问题持续，联系专业维修人员
  
  **注意事项**：
  • 不要自行拆卸外机，避免触电风险
  • 定期清洁外机，保持良好散热
  • 每年进行一次专业维护

**示例2：概念解释**
- **用户**：COP 是什么？
- **系统**：
  **COP (Coefficient of Performance)** 即性能系数，是衡量空调系统能效的重要指标。
  
  **定义**：COP = 制冷量/输入功率
  • 制冷模式：COP_cooling = Q_c / W_net
  • 制热模式：COP_heating = Q_h / W_net
  
  **意义**：
  - COP 值越高，空调系统能效越好
  - 一般来说，COP > 3.0 属于高效系统
  - 影响因素：室外温度、系统维护状态、制冷剂充注量
  
  **优化建议**：定期清洁空调滤网，保持室内外温差合理，避免频繁启停

### 5.5 多轮对话示例

**示例1：连续查询**
- **用户**：Caspian 2021年7月21日的电耗是多少？

- **用户**：这个数值正常吗？
- **系统**：

  • 历史平均值：118.3 kWh
  • 标准差：15.2 kWh
  • 当前值与平均值偏差：+6.1%

**示例2：复杂分析**

## 9. 常见问题排查

### 9.1 端口被占用

```powershell
# 查看 8001 端口
netstat -ano | findstr :8001

# 查看 3000 端口
netstat -ano | findstr :3000

# 结束进程（替换 PID）
taskkill /PID <进程ID> /F
```

### 7.2 Python 模块未找到

```powershell
cd backend
pip install -r requirements.txt
```

### 7.3 npm 依赖问题

```powershell
cd frontend
npm install
```

### 7.4 数据库连接失败

**SQLite**：自动创建，无需配置

**PostgreSQL**：
1. 确认 PostgreSQL 服务运行
2. 确认数据库已创建
3. 检查 `.env` 文件中的密码

### 7.5 数据导入失败

- 检查文件格式是否正确
- 检查文件编码是否为 UTF-8
- 检查数据字段是否符合要求
- 查看系统日志获取详细错误信息

### 7.6 智能问答无响应

- 确认智能助手服务是否运行
- 检查网络连接
- 查看系统日志获取详细错误信息

### 7.7 本地LLM未启动

**问题**：智能问答响应缓慢或无响应
**解决方案**：
1. 确认 Ollama 服务是否运行
2. 检查模型是否已下载：`ollama list`
3. 重新启动 Ollama 服务

### 7.8 打字机效果异常

**问题**：打字机效果卡顿或无响应
**解决方案**：
- 对于长文本（>2000字），系统会自动切换为淡入显示
- 点击停止按钮可截断输出
- 刷新页面重试

### 7.9 中文路径编码问题

**问题**：PowerShell脚本无法启动前端服务
**解决方案**：
- 使用 `restart_services.ps1` 脚本启动服务
- 该脚本已解决中文路径编码问题

### 7.10 系统响应缓慢

**问题**：查询响应时间过长
**解决方案**：
- 对于简单查询（如"XX建筑XX日的电耗"），系统会使用规则引擎（<10ms）
- 对于复杂分析（如"为什么能耗异常"），系统会使用LLM（1-5s）
- 检查服务器资源使用情况
- 优化数据库查询（为常用字段创建索引）
