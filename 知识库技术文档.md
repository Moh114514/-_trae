## 1. 系统概述

### 1.1 什么是知识库系统

BEIMS 知识库系统是一个基于 **RAG（Retrieval-Augmented Generation，检索增强生成）** 架构的智能知识检索引擎。它将建筑能源管理领域的专业知识（国家标准、行业规范、设备手册、运维指南等）向量化存储，在用户提出运维类问题时，自动检索相关文档片段，注入 AI 模型的上下文，生成准确、专业的回答。

### 1.2 解决什么问题

传统建筑能源管理系统面临的问题：

| 痛点 | 知识库方案的解决方式 |
|------|-------------------|
| 运维人员需要翻阅大量国标、行标手册才能找到答案 | 用自然语言提问，系统自动检索并总结 |
| 不同人员的经验水平差异大，回答不一致 | 基于标准文档回答，确保权威性和一致性 |
| 故障排查依赖老员工的经验传承 | 将经验文档化后纳入知识库，形成组织记忆 |
| 新员工培训周期长 | 通过智能问答快速获取知识，降低学习门槛 |

### 1.3 知识库内容范围

当前知识库涵盖以下几类文档：

| 分类 | 文档数量 | 示例 |
|------|---------|------|
| **国家标准（GB）** | ~15 份 | GB/T 19232-2019、GB 50736-2012、GB 55024-2022 |
| **地方标准（DB）** | ~4 份 | DB37/T 5095-2017、DB37/T 5155-2025 |
| **行业标准（JGJ/T）** | ~1 份 | JGJ/T 417-2017 |
| **设备手册** | ~3 份 | KH 系列水源热泵、SLS/SLW 系列水泵 |
| **数据字典** | 1 份 | Data_Dictionary.md |
| **其他文档** | 若干 | 能耗标准、运维指南、FAQ 等 |

## 3. 核心技术栈

### 3.1 技术选型一览

| 组件 | 技术 | 版本/型号 | 作用 |
|------|------|----------|------|
| **向量数据库** | ChromaDB | PersistentClient | 持久化存储文档向量，支持向量相似度检索 |
| **Embedding 模型** | sentence-transformers | paraphrase-multilingual-MiniLM-L12-v2 | 将文本编码为 384 维稠密向量 |
| **LLM 云端模型** | 阿里云 DashScope | qwen-plus | 基于检索到的上下文生成回答 |
| **LLM 本地模型** | Ollama | qwen2.5:7b | 本地意图分析（知识库不可用时降级） |
| **PDF 解析** | pdfplumber | 最新版 | 提取 PDF 文档中的文本内容 |
| **Word 解析** | python-docx | 最新版 | 提取 Word 文档中的文本内容 |
| **模型下载源** | Hugging Face 镜像 | hf-mirror.com | 解决国内网络访问 HuggingFace 的问题 |

### 3.2 Embedding 模型详解

**模型名称：** `paraphrase-multilingual-MiniLM-L12-v2`

| 属性 | 值 |
|------|---|
| **架构** | MiniLM（6 层 Transformer） |
| **语言支持** | 50+ 语言（含中文、英文） |
| **向量维度** | 384 |
| **模型大小** | ~400 MB |
| **训练数据** | 多语言句子对（用于语义相似度任务） |
| **性能特点** | 轻量、快速、多语言效果好 |

**选择理由：**
1. **中文支持好** — 专门针对多语言（含中文）训练，中文语义理解能力强
2. **轻量高效** — MiniLM 架构，推理速度快，内存占用小
3. **适合语义检索** — 专为 paraphrase（语义重写/相似度）任务设计，适合文档块匹配场景
4. **离线可用** — 首次下载后本地缓存，无需联网即可使用

### 3.3 ChromaDB 详解

**ChromaDB** 是一个开源的嵌入式 AI 数据库，专为语义搜索和 RAG 应用设计。

| 属性 | 值 |
|------|---|
| **类型** | 持久化向量数据库（PersistentClient） |
| **存储方式** | 本地文件系统（SQLite + 向量索引） |
| **存储路径** | `E:\openclaw-project\workspace\Fuwu\chroma_db` |
| **集合名称** | `fuwu_knowledge` |
| **相似度度量** | L2（欧氏距离） / Cosine（余弦相似度） |

**存储的数据结构：**

```json
{
  "ids": ["能耗标准_Data_Dictionary.md_0", "能耗标准_Data_Dictionary.md_1", ...],
  "documents": ["文档块 1 的文本内容", "文档块 2 的文本内容", ...],
  "embeddings": [[0.12, -0.34, 0.56, ...], ...],  // 384 维向量
  "metadatas": [
    {
      "source": "Data_Dictionary.md",
      "category": "根目录",
      "path": "E:\\...\\Data_Dictionary.md",
      "chunk": 0,
      "file_hash": "a1b2c3d4e5f6..."
    },
    ...
  ]
}
```

## 4. 知识库构建流程

### 4.1 构建步骤概览

```
Step 1: 初始化
  ├── 加载 Embedding 模型
  └── 连接/创建 ChromaDB

Step 2: 文档扫描
  ├── 遍历 knowledge_docs/ 目录（递归）
  ├── 识别支持的文件格式
  └── 按子目录自动分类

Step 3: 增量检测
  ├── 计算文件 MD5 哈希
  ├── 与已有文档哈希对比
  ├── 未变化的文件 → 跳过
  ├── 已变化的文件 → 删除旧块，标记更新
  └── 新增的文件 → 标记为新增

Step 4: 文档处理
  ├── 读取文件内容（按格式调用不同解析器）
  ├── 按 500 字符/块分割文本
  └── 为每个块生成唯一 ID

Step 5: 向量化存储
  ├── 所有文档块 → Embedding 编码
  └── 批量添加到 ChromaDB

Step 6: 统计输出
  ├── 新增文件数
  ├── 更新文件数
  ├── 删除块数
  ├── 跳过块数
  └── 分类统计
```

### 4.2 文档读取模块

系统支持多种文档格式，每种格式使用专用解析器：

#### Markdown / 纯文本

```python
# 最简单直接，UTF-8 编码读取
with open(filepath, 'r', encoding='utf-8') as f:
    return f.read()
```

#### PDF 文档

```python
import pdfplumber

def read_pdf(filepath):
    text = ""
    with pdfplumber.open(filepath) as pdf:
        for page in pdf.pages:
            page_text = page.extract_text()
            if page_text:
                text += page_text + "\n"
    return text
```

**处理特点：**
- 逐页提取文本，保持页面顺序
- 自动跳过空白页
- 支持扫描件以外的所有 PDF（扫描件需要 OCR，当前不支持）

#### Word 文档

```python
from docx import Document

def read_docx(filepath):
    doc = Document(filepath)
    text = ""
    # 读取段落
    for para in doc.paragraphs:
        text += para.text + "\n"
    # 读取表格内容
    for table in doc.tables:
        for row in table.rows:
            for cell in row.cells:
                text += cell.text + " "
            text += "\n"
    return text
```

### 4.3 文本分块策略

```python
def _split_text(text, chunk_size=500):
    chunks = []
    lines = text.split('\n')
    current_chunk = ""
    
    for line in range:
        if len(current_chunk) + len(line) > chunk_size:
            if current_chunk.strip():
                chunks.append(current_chunk.strip())
            current_chunk = line + "\n"
        else:
            current_chunk += line + "\n"
    
    if current_chunk.strip():
        chunks.append(current_chunk.strip())
    
    return chunks
```

**分块策略说明：**

| 特性 | 说明 |
|------|------|
| **块大小** | 约 500 字符/块 |
| **行级分割** | 不会在行中间截断，保持段落完整性 |
| **无重叠** | 块之间无重叠（未来可引入 overlap 提升上下文连贯性） |
| **空行过滤** | 自动跳过空白块 |

**为什么选择 500 字符？**
- 太小 → 上下文信息不足，检索精度下降
- 太大 → 单次注入 LLM 的 token 过多，成本高且容易丢失关键信息
- 500 字符 ≈ 200-250 中文汉字，包含足够的上下文又不浪费 token

### 4.4 文件哈希与增量更新

**MD5 哈希检测机制：**

```python
def _get_file_hash(filepath):
    hasher = hashlib.md5()
    with open(filepath, 'rb') as f:
        for chunk in iter(lambda: f.read(8192), b''):
            hasher.update(chunk)
    return hasher.hexdigest()
```

**增量更新逻辑：**

```
每次运行 knowledge_base.py 时：

1. 读取 ChromaDB 中已存储的文件哈希列表
2. 扫描 knowledge_docs/ 目录中的所有文件
3. 对每个文件计算 MD5 哈希
4. 对比：
   ├── 哈希一致 → 文件未变化，跳过（节省时间）
   ├── 哈希不同 → 文件已修改，删除旧块，重新分块入库
   └── 新文件 → 分块后直接入库
5. 检测删除：如果文件在目录中不存在但在 ChromaDB 中存在 → 删除对应块
```

**优势：** 知识库文档数量增长后，每次只处理变化的文件，大幅缩短更新耗时。

### 4.5 文档 ID 命名规则

每个文档块使用唯一的复合 ID：

```
{相对路径}_{文件名}_{块序号}

示例：
能耗标准_Data_Dictionary.md_0
能耗标准_Data_Dictionary.md_1
设备手册_KH系列水源热泵_0
设备手册_SLS水泵_2
```

这种命名方式确保：
- 不同文件的块 ID 不冲突
- 同一文件的不同块有序排列
- 子目录结构体现在 ID 中，便于追溯

### 4.6 元数据结构

每个文档块存储以下元数据：

```json
{
  "source": "Data_Dictionary.md",
  "category": "能耗标准",
  "path": "E:\\openclaw-project\\workspace\\Fuwu\\knowledge_docs\\Data_Dictionary.md",
  "chunk": 0,
  "file_hash": "a1b2c3d4e5f6789012345678abcdef01"
}
```

| 元数据字段 | 说明 |
|-----------|------|
| `source` | 文件名，用于来源标注 |
| `category` | 目录分类（子目录名），支持按分类筛选 |
| `path` | 文件完整路径，便于定位原始文件 |
| `chunk` | 块序号（从 0 开始），标记在文档中的位置 |
| `file_hash` | 文件 MD5 哈希，用于增量更新检测 |

## 5. 检索与推理机制

### 5.1 向量检索

当用户提出问题时，系统执行以下步骤：

```python
# 1. 将问题编码为向量
query_embedding = embedder.encode([query])  # → [384 维向量]

# 2. 在 ChromaDB 中搜索最相似的文档块
results = collection.query(
    query_texts=[query],
    n_results=3  # 返回 Top-3 最相关块
)

# 3. 返回结果包含：
# - documents: 匹配的文本块内容
# - metadatas: 对应的元数据（来源、分类等）
# - distances: 相似度距离
```

### 5.2 相似度计算

ChromaDB 使用 **余弦相似度** 衡量问题向量与文档块向量的相似程度：

```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)
```

取值范围：[-1, 1]
- 越接近 1 → 语义越相似
- 越接近 0 → 语义无关

### 5.3 上下文注入

检索到的文档块被拼接为上下文，注入到 LLM 的提示词中：

```python
def build_prompt(query, kb_context, kb_sources):
    prompt = f"""你是建筑能源管理专家。请根据以下数据回答问题。

=== 知识库参考 ===
{kb_context}

=== 用户问题 ===
{query}

要求：专业、简洁、基于数据回答。"""
    
    return prompt
```

**回答中标注来源：**

```python
if kb_sources:
    reply += f"\n\n📚 **知识库来源**：{', '.join(kb_sources)}"
```

## 6. 与对话系统的集成

### 6.1 路由触发机制

知识库检索不是每次对话都触发，而是由**硬规则引擎**检测问题类型后决定是否走知识库路径：

```python
# cloud_edge_router.py — StaticRuleEngine
KNOWLEDGE_KEYWORDS = [
    # 故障处理类
    "怎么处理", "如何处理", "怎么办", "怎么解决", "出了点问题",
    "报警", "异常", "报错", "错误", "故障",
    
    # 设备相关
    "空调", "外机", "冷水机组", "水泵", "风机",
    "设备", "机器", "系统", "机组", "控制器",
    
    # 操作咨询类
    "怎么操作", "如何操作", "怎么做", "操作步骤", "流程",
    
    # 维护保养类
    "维修", "维护", "保养", "检查", "漏水", "漏电", "过热",
    
    # 概念定义类
    "是什么", "什么是", "定义", "概念", "原理", "为什么",
    
    # 标准规范类
    "标准", "规范", "要求", "规定", "参数",
    
    # 优化建议类
    "建议", "优化", "改进", "节能", "省电"
]

def _is_knowledge_query(self, query):
    return any(kw in query for kw in self.KNOWLEDGE_KEYWORDS)
```

### 6.2 路由决策流程

```
用户提问
    │
    ▼
_is_knowledge_query()?
    │
    ├── YES → 第 0 层：知识库路径
    │           1. _search_knowledge(query) → 检索 Top-3 文档块
    │           2. 拼接上下文
    │           3. cloud_llm.process(query, kb_context, kb_sources)
    │           4. 返回回答 + 📚 来源标注
    │
    └── NO → 继续走第 1-4 层路由
                (硬规则 → 语义路由 → 本地推理 → 云端增强)
```

## 7. 技术亮点

### 7.1 🎯 智能触发 — 不浪费算力

知识库检索不是每次对话都执行，而是通过**关键词规则引擎**精准识别需要知识库的问题类型。普通的能耗数据查询走快速 SQL 路径，只有运维/操作/故障类问题才触发知识库检索。

**效果：** 降低不必要的向量检索和 LLM 调用，提升系统整体响应速度。

### 7.2 🔄 热更新 — 零停机更新

知识库支持增量更新，无需清空重建：

| 场景 | 行为 |
|------|------|
| 新增文档 | 自动检测并入库 |
| 修改文档 | MD5 检测变化，删除旧块，插入新块 |
| 删除文档 | 自动从向量库中移除 |
| 未变化的文档 | 跳过处理，节省时间 |

**效果：** 大型知识库（数百文档）更新时，只需处理变化的文件，更新耗时从分钟级降至秒级。

### 7.3 📂 自动分类 — 零配置组织

系统根据文件所在的子目录自动分类：

```
knowledge_docs/
├── 能耗标准/      → category: "能耗标准"
├── 设备手册/      → category: "设备手册"
├── 运维指南/      → category: "运维指南"
└── Data_Dictionary.md  → category: "根目录"
```

支持按分类筛选搜索：

```python
kb.search(query, n_results=3, category="设备手册")
```

### 7.4 🌐 多格式支持 — 不挑文档

支持 Markdown、纯文本、PDF、Word 四种格式，覆盖绝大多数知识文档场景。每种格式使用专用解析器，确保文本提取质量。

### 7.5 🇨🇳 中文优化 — 专为中文场景设计

- **Embedding 模型**：选用多语言模型，中文语义理解能力强
- **HuggingFace 镜像**：配置 `hf-mirror.com` 解决国内下载问题
- **分块策略**：按行分割而非按字符分割，保持中文段落结构

### 7.6 📊 可追溯 — 每个回答都有据可查

每个回答都标注知识库来源：

```
📚 知识库来源：GB 50736-2012.pdf, Data_Dictionary.md
```

用户可以追溯到原始文档，验证回答的准确性。

### 7.7 🔒 离线可用 — 不依赖外部服务

向量数据库和 Embedding 模型都在本地运行：

- ChromaDB → 本地文件系统
- sentence-transformers → 本地加载
- 即使云端 LLM 不可用，知识库的检索功能仍可正常工作

## 10. 扩展与优化建议

### 10.1 已实现的功能

- ✅ 多格式文档支持（MD/TXT/PDF/DOCX）
- ✅ 增量更新（MD5 哈希检测）
- ✅ 自动分类（子目录映射）
- ✅ 按分类筛选搜索
- ✅ 来源标注
- ✅ HuggingFace 国内镜像
- ✅ 智能触发（关键词规则引擎）

### 10.2 建议的优化方向

| 优化项 | 说明 | 优先级 |
|--------|------|--------|
| **块重叠（Overlap）** | 分块时相邻块保留 50-100 字符重叠，避免关键信息被截断 | 高 |
| **混合检索** | 结合向量检索 + BM25 关键词检索，提升精确匹配效果 | 高 |
| **重排序（Rerank）** | 检索后用 Cross-Encoder 对结果重新排序，提升 Top-K 质量 | 中 |
| **多路检索** | 先检索大段落，再在大段落中精确查找 | 中 |
| **Embedding 模型升级** | 升级到 bge-m3 或 GTE 等更强的中文 Embedding 模型 | 中 |
| **OCR 支持** | 增加对扫描件 PDF 的 OCR 识别（Tesseract / PaddleOCR） | 低 |
| **知识库管理 UI** | 提供 Web 界面管理文档（上传、删除、查看统计） | 低 |
| **多语言支持** | 扩展支持英文技术文档 | 低 |

### 10.3 Embedding 模型升级方案

如果希望提升中文检索效果，可以考虑以下模型：

| 模型 | 维度 | 大小 | 特点 |
|------|------|------|------|
| **bge-m3** | 1024 | ~2.2 GB | 目前中文最强，支持多语言、多粒度 |
| **bge-large-zh-v1.5** | 1024 | ~1.3 GB | 专注中文，效果好 |
| **GTE-large-zh** | 1024 | ~1.3 GB | 阿里出品，中文效果好 |
| **当前使用** | 384 | ~400 MB | 轻量快速，够用 |

## 附录 A：RAG 技术名词解释

| 术语 | 全称 | 中文 | 说明 |
|------|------|------|------|
| **RAG** | Retrieval-Augmented Generation | 检索增强生成 | 先检索相关知识，再让 LLM 基于知识生成回答 |
| **Embedding** | — | 嵌入向量 | 将文本转换为数值向量，语义相似的文本向量距离近 |
| **向量数据库** | Vector Database | 向量数据库 | 专门存储和检索高维向量的数据库 |
| **ChromaDB** | — | — | 开源的嵌入式 AI 数据库 |
| **Top-K** | — | 前 K 个 | 检索时返回最相似的前 K 个结果 |
| **余弦相似度** | Cosine Similarity | 余弦相似度 | 衡量两个向量方向相似程度的指标 |
| **Chunk** | — | 文本块 | 将长文档分割为较小的文本片段 |
| **MD5** | Message-Digest Algorithm 5 | 消息摘要算法5 | 用于检测文件是否变化的哈希算法 |
